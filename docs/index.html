The Problem We Solve

FastAPI is great for Day 1.
Day 100 is where it fails you.

When your app grows to 150 endpoints and 8 domains, you need more than a fast router. You need enforced architecture, not just recommended patterns.

❌ Without Pillar

Architecture by convention

Routers importing repositories, circular dependencies between services, background tasks scattered across a Celery worker that needs its own Redis, its own Docker container, and its own deployment config.

✅ With Pillar

Architecture by enforcement

An AST-level import checker blocks layer violations at startup. The built-in SQLite WAL task queue handles background work with zero extra infrastructure. DI resolves your entire dependency graph from type hints alone.

❌ The old way

Three stacks for one feature

Send a welcome email → write a Celery task, configure a Redis broker, run a separate worker process, add Flower for monitoring, debug why tasks are silently dropped on server restart.

✅ The Pillar way

One decorator, zero infrastructure

Add @background_task(retries=3) to any method. Pillar serialises the call to a persistent SQLite queue. Tasks survive restarts. When you need scale, swap driver to Redis in pillar.toml.

Core Features

Everything you need. Nothing you don't.

Pillar is opinionated so your team doesn't have to be. Every feature is built-in, not bolted on.

⚡
Rust-Powered Router

The matchit radix-tree router compiled to native code via PyO3. Route matching at C-speeds — not Python regex speeds.

🏗️
Architecture Enforcement

AST-level DDD import checker. If router.py imports from repository.py, the server refuses to start. Rules, not recommendations.

📦
Built-in Task Queue

Persistent SQLite WAL queue backed by the Rust engine. Tasks survive restarts. Retry on failure. No Redis, no separate worker, no Flower dashboard.

🔌
Zero-Boilerplate DI

Type-hint a handler parameter with a class and Pillar recursively resolves the entire dependency graph. No Depends(). No factories. No XML config.

🌉
Smart Async/Sync Bridge

Write sync DB code in repositories, async I/O in services — the Smart Bridge dispatches sync calls to a thread pool via run_in_executor automatically.

📡
Native WebSockets

First-class WebSocket support with @router.websocket(). Full DI injection works identically to HTTP handlers — no special setup required.

📋
OpenAPI 3.1 Out of the Box

Auto-generated OpenAPI schema, Swagger UI at /docs, ReDoc at /redoc, and an interactive guide at /guide — from day one.

✅
Production Built-ins

Health probes, Prometheus-format metrics, CORS, security headers, request IDs, and response-time tracking — all enabled by default, all configurable.

The enforced layer model

Every Pillar domain follows the same three-layer structure. The framework's AST checker ensures the rules hold as the codebase grows.

  • Repository — raw DB queries only. May only import schemas.
  • Service — business logic. May import repository and schemas.
  • Router — HTTP definitions. May import service and schemas.
Build your first domain →
domains/users/service.py python 
from pillar import background_task
from pillar.exceptions import NotFoundError
from .repository import UserRepository   # ✓ allowed

class UserService:
    def __init__(self, repo: UserRepository):
        self.repo = repo   # auto-injected, no Depends()

    def get_user(self, user_id: int) -> dict:
        user = self.repo.get_by_id(user_id)
        if not user:
            raise NotFoundError(f"User {user_id} not found")
        return {**user, "status": "active"}

    def create_user(self, name: str, email: str) -> dict:
        user = self.repo.create(name, email)
        self.send_welcome_email(user["email"])  # queued!
        return {**user, "status": "active"}

    @background_task(retries=3)
    def send_welcome_email(self, email: str):

Philosophy & Learnings

What building Pillar taught us

Five hard-won insights from watching Python backends scale beyond day 100.

1

Clean Architecture must be enforced, not recommended

Pattern discipline erodes under deadline pressure. The only reliable solution is a compiler-level check that runs before the first request is ever served.

2

Background tasks don't need dedicated infrastructure

For 90% of workloads, an embedded SQLite WAL queue is faster, simpler, and more reliable than Celery + Redis. Complexity should scale with actual need.

3

Rust and Python can coexist without friction

PyO3 gives you native-speed primitives — radix-tree routing, persistent queues, type contracts — with zero Python boilerplate on the hot path.

4

DI without decorators is possible and cleaner

Type hints are already your contract. Respecting them at resolution time — not through wrapper decorators — keeps handler code readable and testable.

5

Production-readiness is a Day 1 decision

Health checks, Prometheus metrics, CORS, and security headers should be on by default. Opting out is easier than retrofitting them six months after launch.

6

Sync and async should coexist naturally

Forcing every DB query to be async just to avoid blocking the event loop is noise. Let the framework handle the dispatch — developers handle the logic.
