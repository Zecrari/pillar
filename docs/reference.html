📋 Reference

Complete reference for the Pillar CLI, built-in HTTP endpoints, all pillar.toml configuration keys, exceptions, and Docker deployment.

CLI — pillar run #

pillar run Start the ASGI server (uvicorn under the hood)
  • APP — ASGI application path, e.g. main:app (required)
  • --host TEXT — Bind host (default: 0.0.0.0)
  • --port INTEGER — Bind port (default: 8000)
  • --reload — Auto-reload on file changes (development only)
  • --workers INTEGER — Number of worker processes (default: 1)
  • --log-level TEXT — Logging level: debug|info|warning|error|critical
examplesbash
# Development — with hot reload
pillar run main:app --reload

# Production — 4 workers, port 80
pillar run main:app --workers 4 --port 80 --log-level warning

# Custom host
pillar run main:app --host 127.0.0.1 --port 5000

CLI — pillar new #

pillar new <name> Scaffold a new Pillar project in a new directory
  • NAME — Project name / directory (required)
  • --template TEXT — Starter template: minimal|api|fullstack (default: api)
bashbash
pillar new my-api
cd my-api
pillar run main:app --reload

Generated structure:

├── requirements.txt
└── domains/

        ├── repository.py
        └── schemas.py

CLI — pillar generate #

pillar generate dockerfile Write a production-ready Dockerfile to ./Dockerfile
pillar generate domain <name> Scaffold a new DDD domain (router/service/repository/schemas)
bashbash

# → writes Dockerfile to ./Dockerfile

pillar generate domain orders
# → creates domains/orders/{router,service,repository,schemas}.py

Debug Endpoints #

⚠ Debug mode only

These endpoints are only registered when debug = true in pillar.toml. They are never available in production (the framework checks app.debug at startup, not at request time).

EndpointDescription
GET /debug/routesRegistered routes with full handler paths
GET /debug/diDI container — registered classes and resolved singletons
GET /debug/queueQueue state — pending, running, failed task counts
GET /debug/configActive merged config (TOML + env vars, secrets redacted)

Environment Variable Overrides #

Any config key can be overridden with an environment variable using the pattern PILLAR_<SECTION>_<KEY> (uppercased). Env vars take precedence over pillar.toml.

Env VariableTOML equivalent
PILLAR_APP_DEBUG[app] debug
PILLAR_APP_ENV[app] env
PILLAR_DATABASE_URL[database] url
PILLAR_QUEUE_DRIVER[queue] driver
PILLAR_QUEUE_REDIS_URL[queue] redis_url
PILLAR_CORS_ALLOW_ORIGINS[cors] allow_origins
PILLAR_DOCS_ENABLED[docs] enabled
.env / Docker / CIbash
export PILLAR_APP_DEBUG=false
export PILLAR_DATABASE_URL=sqlite:///prod.db
export PILLAR_QUEUE_DRIVER=redis
export PILLAR_QUEUE_REDIS_URL=redis://redis:6379/0
export PILLAR_DOCS_ENABLED=false    # hide API docs in production

ForbiddenError403Credentials valid but the user lacks permission
ValidationError422Input data fails business-rule validation (not just schema)

from pillar.exceptions import (
    PillarError,               # base class for all Pillar exceptions

usage in service.pypython
from pillar.exceptions import NotFoundError, ConflictError, ValidationError

class UserService:
    def get(self, user_id: int):
        user = self.repo.get(user_id)
        if not user:
            raise NotFoundError(f"User {user_id} not found")
        return user

    def create(self, data: UserCreate):
        if self.repo.email_exists(data.email):
            raise ConflictError(f"Email already registered: {data.email}")
        if not data.password or len(data.password) < 8:
            raise ValidationError("Password must be at least 8 characters")
        return self.repo.create(data)

    def delete(self, requester_id: int, target_id: int):
        requester = self.repo.get(requester_id)
        if requester["role"] != "admin":
            raise ForbiddenError("Only admins can delete users")

Custom Exceptions #

exceptions.pypython
from pillar.exceptions import PillarError

class PaymentDeclinedError(PillarError):
    status_code = 402
    default_message = "Payment was declined by the processor"

class RateLimitError(PillarError):
    status_code = 429
    default_message = "Too many requests"

    def __init__(self, retry_after_seconds: int = 60):
        super().__init__(self.default_message)
        self.retry_after = retry_after_seconds

    def to_response(self):
        return {
            "error": str(self),
            "retry_after": self.retry_after
        }, self.status_code

# Raise anywhere in the service / repository layer
raise PaymentDeclinedError("Insufficient funds")
raise RateLimitError(retry_after_seconds=30)

Generate Dockerfile #

bashbash
pillar generate dockerfile
# → writes the following to ./Dockerfile
Dockerfile (generated)bash
FROM python:3.11-slim AS base

WORKDIR /app

# Install system dependencies for Rust-compiled extension
RUN apt-get update \
    && apt-get install -y --no-install-recommends gcc \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

# Expose port
EXPOSE 8000

# Health check
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

# Run with multiple workers
CMD ["pillar", "run", "main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]

Production Checklist #

Checklist ItemHowWhy
Disable debugPILLAR_APP_DEBUG=falseHides /debug/* endpoints
Disable API docsPILLAR_DOCS_ENABLED=falseDon't expose schema to public
Restrict CORSallow_origins = ["https://yourdomain.com"]Prevent cross-origin attacks
Use multiple workers--workers 4 (or 2 × CPU + 1)Parallelism under load
Mount a persistent volumeMap /app/data → host or cloud volumeSQLite survives container restarts
Wire up health checksKubernetes/ECS liveness: GET /healthAuto-restart on deadlock
Use readiness for trafficReadiness: GET /readyOnly send traffic when DB/queue ready
Pin Python versionFROM python:3.11-slimReproducible builds
💡 SQLite in production

SQLite in WAL mode easily handles 10 000+ req/s for read-heavy workloads on a single server. It's a valid production choice unless you need multiple write replicas. When you outgrow it, update PILLAR_DATABASE_URL — the repository layer is unchanged.
